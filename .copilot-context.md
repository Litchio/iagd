# Grim Dawn Item Assistant (GDIA) - Linux/Tauri Port Handover

## Project Overview
We are porting the Windows-only tool "Grim Dawn Item Assistant" to a native Linux environment (Bazzite/SteamOS) using a hybrid architecture.

- **The Game:** Grim Dawn (x64) running via Steam Proton.
- **The Proxy DLL (C++):** A custom `dinput8.dll` cross-compiled via MinGW-w64 on Ubuntu. It uses Microsoft Detours to hook game functions, intercepts dropped items, and sends them as JSON over Winsock2 TCP to `127.0.0.1:1337`.
- **The Backend (Rust/Tauri):** A native Linux tokio TCP server listening on port 1337, parsing JSON, and storing items in a local SQLite database.
- **The Frontend (React):** Existing React UI served via Tauri (WebKitGTK) natively on Linux.

## Current Blocker: The Proton Black Screen

### How to Reproduce
1. Place compiled DLL: `HookDll/Hook/build/bin/dinput8.dll` → `/path/to/Grim Dawn/x64/dinput8.dll`
2. Launch via Steam with: `WINEDLLOVERRIDES="dinput8=n,b" %command%`
3. Game starts but hangs on black screen; audio may still play
4. Proton logs confirm the DLL was loaded and `DLL_PROCESS_ATTACH` fired

### Symptom
- Game loads DLL successfully
- No immediate crash (segfault) observed in initial logs
- UI renders but hangs (black screen, input unresponsive)
- Possibly the game's graphics initialization or input handling is blocked

### What We Already Fixed (Attempts 1-3)

#### Fix 1: DirectInput8Create Export
**Problem:** Game called `LoadLibrary("dinput8.dll")` but then couldn't find `DirectInput8Create` export, causing immediate crash.

**Solution:** Added `extern "C" __declspec(dllexport)` export that:
- Loads real `C:\Windows\System32\dinput8.dll` via `LoadLibraryA()`
- Gets the real `DirectInput8Create` function pointer via `GetProcAddress()`
- Forwards all calls to the real implementation
- Returns `E_FAIL` if loading fails

**File:** `HookDll/Hook/dllmain.cpp` (lines ~560-610)

**Status:** ✅ Works - export is present and callable. Verified via `x86_64-w64-mingw32-nm dinput8.dll | grep DirectInput8Create`

#### Fix 2: Loader Lock Deadlock (TCP Connection)
**Problem:** `ProcessAttach()` was calling `g_tcpClient->Connect()` synchronously, blocking `DllMain` and causing loader lock deadlock.

**Solution:** Wrapped TCP connection in a detached `std::thread`:
```cpp
std::thread tcpConnectThread([]() {
    try {
        if (!g_tcpClient->Connect()) {
            LogToFile(LogLevel::WARNING, L"Failed to connect to TCP server");
        }
    }
    catch (...) {
        LogToFile(LogLevel::WARNING, L"Exception during TCP connection");
    }
});
tcpConnectThread.detach();  // Returns immediately
```

**File:** `HookDll/Hook/dllmain.cpp` (lines ~500-515)

**Status:** ✅ DllMain returns immediately. TCP connection happens asynchronously.

#### Fix 3: Premature DLL Unload (DLL Pinning)
**Problem:** Proton logs showed that Grim Dawn calls `LoadLibrary("dinput8.dll")` to probe for controllers, then immediately calls `FreeLibrary()`. This unloaded our DLL while the detached TCP thread was still running, causing an Access Violation (segfault) when the thread tried to execute code no longer in memory.

**Solution:** Pin the DLL in memory using `GetModuleHandleEx` in `ProcessAttach()`:
```cpp
HMODULE hDummy = NULL;
if (GetModuleHandleExA(GET_MODULE_HANDLE_EX_FLAG_FROM_ADDRESS | GET_MODULE_HANDLE_EX_FLAG_PIN, 
                       (LPCSTR)&ProcessAttach, &hDummy)) {
    LogToFile(LogLevel::INFO, L"DLL pinned in memory successfully");
}
else {
    // Fallback: bump reference count via LoadLibrary
    LoadLibraryA("dinput8.dll");
    LogToFile(LogLevel::INFO, L"Fallback DLL pinning via LoadLibrary completed");
}
```

**File:** `HookDll/Hook/dllmain.cpp` (lines ~418-432)

**Status:** ✅ DLL stays loaded for entire game session, even if game calls `FreeLibrary()`.

### Still Broken: Unknown Cause
Despite all three fixes, the game still black-screens. The DLL loads, doesn't segfault, but something hangs the game.

## Hypotheses for the Black Screen (Next Steps for New Agent)

### Hypothesis 1: Detours Hooks Crashing Silently
**Concern:** We replaced MSVC `__try/__except` with a MinGW-compatible macro mapping to C++ `try/catch(...)`. This might not handle all edge cases, or the compatibility shim might not be working correctly.

**Evidence:**
- No segfault in logs suggests graceful exception handling
- But the exception might be silently swallowing a crash

**Debug Steps:**
- Add extensive file logging to every hook enable/disable
- Log the exact address being hooked in `OnDemandSeedInfo`, `InventorySack_AddItem`, etc.
- Log the result of `DetourTransactionBegin()`, `DetourAttach()`, `DetourTransactionCommit()`

### Hypothesis 2: Winsock2 Initialization Failing in Detached Thread
**Concern:** The detached TCP thread calls `g_tcpClient->Connect()` which internally initializes Winsock2 via `WSAStartup()`. Under Proton/Wine, this might fail or block.

**Evidence:**
- No error logged if `Connect()` fails silently
- The thread might be hanging on socket operations

**Debug Steps:**
- Add logging to `TcpClient::Connect()` before and after every Winsock call
- Log `WSAStartup()` result and error codes
- Log socket creation (`socket()`) success/failure
- Log `connect()` attempt and result
- Check if thread is actually detaching or being waited on somewhere

### Hypothesis 3: DirectInput8Create Export Forwarding Malformed
**Concern:** The export forwarding might be loading the wrong DLL or causing infinite recursion.

**Evidence:**
- The game doesn't crash immediately, suggesting the export is callable
- But the game might enter an infinite loop or deadlock in export forwarding

**Debug Steps:**
- Add extensive logging in `DirectInput8Create()` export function
- Log when called, what DLL is loaded, function pointer obtained
- Log when forwarding to real function and result
- Verify we're not recursively calling our own `DirectInput8Create()`

### Hypothesis 4: Game Graphics/Input Initialization Blocked
**Concern:** The hooks might be interfering with game initialization that happens after DLL load.

**Evidence:**
- DLL loads successfully but game hangs before rendering
- Audio plays (suggests game is running, just UI frozen)

**Debug Steps:**
- Add checkpoint logging throughout `ProcessAttach()` to find where it hangs
- Log before and after every `EnableHook()` call
- Log OnDemandSeedInfo creation and initialization
- Log Worker thread startup
- Narrow down which hook (if any) is causing the hang

### Hypothesis 5: Settings.json Missing or Malformed
**Concern:** `SettingsReader()` might crash if `settings.json` doesn't exist or has syntax errors.

**Evidence:**
- Caught exception in ProcessAttach but error logged might not be visible

**Debug Steps:**
- Ensure `IAGrim/settings.json` exists in the game directory
- Log the full JSON parsing result
- Add try-catch around every JSON field access
