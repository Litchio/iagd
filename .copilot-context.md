# Grim Dawn Item Assistant (GDIA) - Linux/Tauri Port Handover (v2)

## Project Overview
Porting the Windows-only tool "Grim Dawn Item Assistant" to native Linux (Bazzite/SteamOS).

- **Game:** Grim Dawn x64 via Steam Proton (public/default branch, latest version).
- **Proxy DLL (C++):** custom `dinput8.dll` cross-compiled with MinGW-w64. Hooks game functions via Microsoft Detours, intercepts looted items, sends JSON over Winsock2 TCP to `127.0.0.1:1337`.
- **Backend (Rust):** plain Cargo binary `gdia-tauri/` (NOT a Tauri app yet, despite the name). Tokio TCP server on port 1337 → SQLite (`gdia-tauri/data/gdia.db`). Sources: `src/main.rs`, `tcp_server.rs`, `database.rs`, `models.rs`. Built binary: `target/debug/gdia_backend`.
- **Frontend (React):** `WebUI/` — integration pending (Phase 3).

## Root-cause findings from code review (2026-07-23)

The previous handover listed the black screen as "unknown root cause". Static analysis found these defects, several individually sufficient to cause the symptom:

1. **CRT static-init NULL landmines (highest confidence).** `GrimTypes.h` had ~12 `static auto fn* = ...(GetProcAddressOrLogToFile(...))` per-TU statics and `GrimTypes.cpp` had 4 globals, all resolving `game.dll`/`engine.dll` exports at DLL load time, before `DllMain`. If Proton loads the proxy before the game DLLs, every pointer is NULL; `fnGetGameEngine()` then dereferenced `*(DWORD_PTR*)NULL` → access violation during process init.
2. **Render loop was hooked from frame 0.** `OnDemandSeedInfo::EnableHook()` detours `Engine::Render`; `InventorySack_AddItem` detours `GameEngine::Update`. Both hooks called game APIs (NULL after #1) on the render/game threads every frame → black screen with audio continuing (exact reported symptom). `m_isActive` was also never initialized in the ctor.
3. **MinGW SEH shim is semantically wrong.** `detours_mingw_compat.h` maps `__try/__except` → `try/catch(...)`, but `catch(...)` CANNOT trap access violations (SEH). Any fault inside Detours' memory-probing code would be an unhandled exception (often under loader lock), not a graceful failure.
4. **DirectInput8Create recursion hazard.** If loading `C:\Windows\System32\dinput8.dll` failed, the fallback `LoadLibraryA("dinput8.dll")` could resolve back to OUR OWN module → infinite recursion → stack overflow. Also the DLL exported ONLY `DirectInput8Create`; real dinput8 also exports the COM entry points.
5. **Logging was silently dead on a fresh prefix.** `HookLog` never created `%appdata%\..\local\evilsoft\iagd\`; if missing, `m_out.open()` failed and ALL logs vanished. Plus static-init-order UB: `GrimTypes` statics called `LogToFile` before `g_log` was necessarily constructed.
6. **Wine mode required settings.json.** Without `persistent.isRunningInWine=true` in the prefix, the linuxhack IPC folder / PID file logic never ran.
7. **No Detours return codes checked anywhere**; `DetourAttach` was called even on NULL targets (three raw attach blocks in `InventorySack_AddItem::EnableHook`).
8. **Old init aborted instead of waiting.** `ProcessAttach` returned FALSE if the engine wasn't online yet — fatal when the DLL loads early via `WINEDLLOVERRIDES` static import.

## Changes made (Phase A: observability, Phase B: landmine removal)

- **`DebugLog.h` (new):** loader-lock/CRT-safe logger, Win32-only (CreateFileA/WriteFile per line, unbuffered), writes **`dinput8_debug.log` next to the DLL** (game `x64/` folder). Macro: `DBGLOG(fmt, ...)`. Timestamps, pid/tid per line.
- **`HookLog.cpp`:** creates the log directory (`std::filesystem::create_directories`) before opening; NULL-safe `SHGetKnownFolderPath`.
- **`dllmain.cpp`:**
  - `g_log` → construct-on-first-use (`GetHookLog()` + macro) — fixes init-order UB.
  - `ProcessAttach` is now minimal: loads hook config, auto-detects Wine (`ntdll!wine_get_version` — no settings.json needed), writes PID file, pins DLL, then spawns a **detached `DeferredInit` thread** and returns TRUE immediately. Nothing heavy runs under the loader lock anymore.
  - `DeferredInit`: waits (poll, with timeouts) for `game.dll`+`engine.dll` → `ResolveGameApi()` → waits for engine online → creates event/hooks → enables hooks → TCP connect thread → worker thread. Failures log + `ReportCancelledInjection` instead of crashing.
  - `DirectInput8Create`: cached real-module handle, **self-recursion guard** (refuses to forward to own module), full DBGLOG instrumentation; added `DllGetClassObject`/`DllCanUnloadNow`/`DllRegisterServer`/`DllUnregisterServer` exports forwarding to the real DLL.
- **`GrimTypes.h/.cpp`:** all function pointers are now `extern ... = nullptr`, filled by **`ResolveGameApi()`** (non-blocking, idempotent, logs every failure with the mangled name — if a game update renames exports, the log says exactly which one). `fnGetGameEngine`/`fnGetEngine` NULL-check before dereferencing the global. `IsGameApiResolved()` added.
- **`BaseMethodHook.cpp`:** `HookDll` checks GetProcAddress result (skips NULL targets) and ALL Detours return codes (`DetourTransactionBegin/UpdateThread/Attach/Commit`), DBGLOGs target address + trampoline; `Unhook` NULL-guard + checked.
- **`HookConfig.h/.cpp` (new):** runtime hook matrix via **`dinput8_config.json`** next to the DLL. Keys (all default `true` except `passthroughOnly` default `false`): `passthroughOnly`, `hookInstaloot`, `hookGameEngineUpdate`, `hookEngineRender`, `startSeedInfoThread`, `startTcpClient`.
- **`OnDemandSeedInfo.cpp`:** `m_isActive=false` in ctor; `EnableHook` gated on `hookEngineRender`; `Hooked_Engine_Render` + `HookedGameSetDifficultyRampMethod` have first-call DBGLOG one-shots and full NULL/`IsGameApiResolved()` guards (fall through to original render).
- **`InventorySack_AddItem.cpp`:** `GameEngine::Update` hook gated on `hookGameEngineUpdate`; the three raw Detours blocks (GameInfo ctor, AddItem_Drop, AddItem_Vec2) now route through the checked `HookDll`; `Hooked_GameEngine_Update` guarded + first-call log.
- **`Detours-master/src/detours.cpp`:** `detour_is_imported` (the one SEH probe on the attach path) rewritten with `VirtualQuery` page validation instead of `__try/__except`. (`modules.cpp` still has shimmed probes but is dead code for our call path.)
- **`CMakeLists.txt`:** added `HookConfig.cpp`.

## Phase C: on-device test matrix (bisect the black screen)

Deploy: `cp HookDll/Hook/build/bin/dinput8.dll "<steamapps>/common/Grim Dawn/x64/dinput8.dll"`

Steam launch options:
```
PROTON_LOG=1 PROTON_LOG_DIR=/home/deck/gdia-logs WINEDEBUG=+loaddll,+seh WINEDLLOVERRIDES="dinput8=n,b" %command%
```

Logs to collect per run:
1. `<steamapps>/common/Grim Dawn/x64/dinput8_debug.log` — our checkpoint log (always works)
2. `/home/deck/gdia-logs/steam-219990.log` — Proton log (load order, unhandled exceptions)
3. `<steamapps>/compatdata/219990/pfx/drive_c/users/steamuser/AppData/Local/evilsoft/iagd/iagd_hook.log` — hook log (now reliably created)

Run tests **in order**, one `dinput8_config.json` per test (place next to the DLL), stop at the first config that changes behavior:

| Test | config | Meaning if game works / still black |
|------|--------|-------------------------------------|
| 0 | `{"passthroughOnly": true}` | works → a hook/thread causes it. black → proxy/forwarding itself is broken (check `DirectInput8Create` lines in debug log) |
| 1 | no config file (defaults) | all hooks on - baseline |
| 2 | `{"hookEngineRender": false, "startSeedInfoThread": false}` | works → the render hook is the culprit |
| 3 | `{"hookGameEngineUpdate": false}` (render on) | works → the update hook is the culprit |
| 4 | `{"startTcpClient": false}` | works → Winsock/connect issue |

Interpretation guide for `dinput8_debug.log`: expect sequence `CRT static initializers` → `ProcessAttach entered` → `Wine auto-detect: YES` → `DLL pinned` → `DeferredInit: thread started` → `game.dll=... engine.dll=...` → `game API resolved` → `engine online=1` → `enabling hook N` → `HookDll: SUCCESS hooking ...` → `FIRST CALL - render/update hook is live`. The last line before a hang localizes the fault.

Note: hooks resolve by mangled export NAME, so game patches only break us if Crate changes a hooked function's signature; `ResolveGameApi: FAILED to resolve <name>` lines identify that immediately.
